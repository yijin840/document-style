# document-style
给自己的文档整理一套规范

> 针对开发、技术、需求、业务之类的文档而言
>
> 写文档不能太死板，避免出现 AI 风格，和过于机械化的风格，所以要求如下：

1. 要交代历史背景、痛点、选型对比，把痛点都量化出来，要有具体的数值进行支撑，尽量说明原因而不是仅给结果。
2. 避免空洞词汇，代入具体的数值指标，描述具体的异常场景。
3. 打破对齐感，核心部分增加要点，注释和提示等部分。
4. 多使用主动语态，侧重讲解和解释，让人看懂，而不是单纯汇报工作。
5. 不要只依赖 1.1、1.2、1.3 这种死板结构，也可根据模块进行划分。
6. 一定要结合真实的场景和内部的问题，不能脱离实际去编写。
   1. 业务背景：描写存在的瓶颈，以及为什么会出现这个业务。
   2. 技术选型：描述架构变化，根据具体问题说明架构从 v1 -> v2 的演进，v1 的优劣以及 v2 的改进。
   3. 接口定义：字段要有具体数值、取值范围、约束条件和错误码标识，类型和异常要有统一处理，把具体的 JSON 格式数据贴出来直观展示。
   4. 测试阶段：不能只给单一测试描述，所有测试包括功能测试、回归测试、性能测试、安全测试，并给出具体测试流程、测试重点、数据模板及异常处理。
7. 多采用图文结合的形式，便于阅读者有更深刻的理解。
   1. 架构图和流程图：标注具体关键组件和数据流向。
   2. 状态机/时序图：接口要有具体状态变化和调用顺序。
   3. 标注图上的瓶颈点和失败点。
