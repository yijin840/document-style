# document-style
给自己的文档整理一套规范

> 针对开发、技术、需求、业务之类的文档而言
写文档不能太死板，避免出现AI风格，和过于机械化的风格，所以要求如下：
1.  要交代历史背景、痛点、选型对比，把痛点都量化出来，要有具体的数值进行支撑，尽量将原因而不是最后的结果
2.  避免空洞词汇，代入具体的数值指标，描述具体的异常场景
3.  打破对齐感，核心部分增加要点，注释和提示等部分
4.  多使用主动语态，侧重于讲解，解释，给人看懂，而不是汇报工作
5.  不要只依赖1.1, 1.2, 1.3这种死板的结构，也可根据模块进行划分
6.  一定要结合真实的场景和内部的问题，不能脱离实际的去编写
  a. 业务背景，描写存在的瓶颈等，以及为什么出现这个业务
  b. 技术选型，描述架构变化，根据具体存在的问题，架构从v1->v2的变迁，v1的优劣，v2的改进
  c. 接口定义，字段要有具体的数值，取值范围，约束条件和错误码标识，类型和异常要有统一的处理，把具体的json格式数据贴出来进行直观的展示
  d. 测试阶段，不能给出单一的测试描述，所有的测试包括功能测试、回归测试、性能测试、安全测试，还要有具体的测试流程，测试的重点，数据模版之类，及具体的异常处理
7. 多采用图的形式和文字描述进行结合，便于阅读者有更深刻的理解
  a. 架构图和流程图， 标注出来具体的关键组件，数据流向
  b. 状态机/时序图， 接口要有具体的状态变化，调用顺序
  c. 标注出来图上瓶颈点和失败点
